# 多风格音频键盘映射播放器需求规格说明书

## 1. 项目概述

### 1.1 项目背景
基于B站视频《【混合风格】Doppelganger中盘旋律多风格Mix》的创作理念，开发一个本地音频采样器程序。该视频将原曲《Doppelganger》的中间旋律部分改编为11种不同曲风，本程序将这些音频片段映射到电脑键盘上，实现类似钢琴或MIDI键盘的实时演奏体验。

### 1.2 项目目标
开发一个轻量级、低延迟的音频采样播放器，支持：
- 将11个等长音频片段映射到键盘按键
- 实时触发播放，支持多键同时按下（复音）
- 流畅的音频切换和混合效果

### 1.3 目标用户
- 音乐创作者和DJ
- 音游爱好者
- 现场表演者
- 音乐实验爱好者

## 2. 功能需求

### 2.1 核心功能

#### 2.1.1 音频加载与管理
- **功能描述**：加载11个等长的音频文件作为采样
- **支持格式**：MP3、WAV、FLAC
- **音频要求**：
  - 所有音频必须等长（时长一致）
  - 推荐时长：2-8秒
  - 采样率：44.1kHz 或 48kHz
- **加载方式**：
  - 批量选择11个文件
  - 拖拽文件夹（按文件名排序自动加载）
  - 单独替换某个键位的音频

#### 2.1.2 键盘映射系统

**默认键盘映射**：

| 键位 | 风格编号 | 显示标识 |
|------|----------|----------|
| A | 风格1 | 01 |
| S | 风格2 | 02 |
| D | 风格3 | 03 |
| F | 风格4 | 04 |
| G | 风格5 | 05 |
| H | 风格6 | 06 |
| J | 风格7 | 07 |
| K | 风格8 | 08 |
| L | 风格9 | 09 |
| ; | 风格10 | 10 |
| ' | 风格11 | 11 |

**映射功能**：
- 支持自定义键盘映射
- 支持MIDI键盘输入（可选扩展）
- 按键触发模式（默认：门控模式）：
  - **门控模式**（Gate）⭐默认：按住键时输出该风格，松开键时静音（播放头暂停）。切换按键时播放头连续前进，新风格从当前位置接续
  - **触发模式**（Trigger）：按下键后持续输出该风格直到手动切换或音频结束，无需按住。适合快速切换场景
  - **循环模式**（Loop）：播放头到达音频末尾后自动回到开头循环，适合无限即兴演奏

#### 2.1.3 实时播放引擎（共享时间线模型）

**核心概念**：

11个音频片段是同一旋律的不同风格编排，时间完全对齐。程序维护一个**全局播放头（Playhead）**，所有风格共享同一条时间线。按键的作用是**选择当前输出哪个风格的音频**，而非独立触发播放。

**播放模型示意**：

```
时间线:  0s ─────── 1s ─────── 2s ─────── 3s ─────── 4s ───────
风格1:   [████████████████████]
风格2:   [                ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░]
风格3:   [                                ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒]
         ↑                ↑                ↑
       按下A            松开A,按下S       松开S,按下D
       播放风格1        播放风格2(从2s起) 播放风格3(从4s起)

播放头: ──────────────────────────────────────────────────────→
```

**播放规则**：

1. **首次按键**：播放头从0秒开始，播放该键对应的风格音频
2. **切换按键**：松开当前键、按下新键时，播放头**不回退**，新风格从播放头当前位置继续播放
3. **松开所有键**：播放头暂停，记住当前位置；下次按下任意键时从暂停位置继续
4. **播放到末尾 — 循环回跳**：播放头到达音频结尾时，检测当前是否有按键处于按下状态：
   - **有按键按下**：播放头自动跳转到**循环起点**（默认为音频开头 0s），继续播放当前风格
   - **无按键按下**：播放头暂停在末尾，等待下次按键
5. **无键按下时**：静音，播放头不前进

**循环回跳设置**：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| 循环起点 | 播放到末尾后回跳的时间位置 | 0.0s（音频开头） |
| 循环模式 | 是否启用自动循环（关闭/按键保持时循环/始终循环） | 按键保持时循环 |
| 循环过渡 | 回跳时的过渡方式（硬切/淡入淡出） | 淡入淡出（50ms） |

循环起点设置方式：
- **手动输入**：直接输入回跳的时间点（秒）
- **节拍定位**：输入节拍编号，自动计算对应时间（如"第8拍"→ 8 × 60/BPM 秒）
- **播放头定位**：在波形上点击设置循环起点标记

**核心特性**：
- **低延迟**：< 20ms 按键到切换响应延迟
- **时间对齐**：所有风格在同一时间点的内容完全对应（旋律、节拍一致）
- **无缝切换**：风格切换时音频时间连续，无跳跃或断裂
- **混合输出**：支持短时间重叠（交叉淡化），避免切换时的爆音

**播放控制**：
- 全局音量控制（0-100%）
- 单个风格音量调节（0-150%，支持增益）
- 全局播放速度调节（0.5x - 2.0x，保持音调）
- 全局音调调节（-12到+12半音）
- 播放头位置显示与手动拖拽定位
- 重置播放头到开头

#### 2.1.4 演奏模式

**模式一：自由演奏模式**
- 按下任意键选择当前输出的风格，播放头从当前位置前进
- 切换按键时风格无缝衔接，播放头不回退
- 松开所有键时播放头暂停，再次按键从暂停处继续
- 实时显示当前播放头位置和激活的风格

**模式二：序列录制模式**

录制与回放机制：

- **录制内容**：记录风格切换事件序列，包括：
  - 切换到的风格编号
  - 切换时间点（相对于录制起始点的播放头位置）
  - 该风格持续时长（从按下到松开/切换的时间）
- **录制数据格式**：每个事件存储为 `(style_index, start_time, duration)` 三元组列表
- **录制示例**：
  ```
  事件序列: [(0, 0.0, 2.0), (3, 2.0, 1.5), (7, 3.5, 0.5), ...]
  含义: 风格1从0s播放到2s → 风格4从2s播放到3.5s → 风格8从3.5s播放到4s
  ```

播放时长保证机制：

由于采用**门控模式**，每次按键的实际持续时长取决于用户按住的时间，手动演奏时必然存在时长偏差。在共享时间线模型下，回放时需要保证每个风格的播放时长准确，以维持整体时间线的连续性。提供以下三种量化策略：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **原始时长** | 完全按照录制时的持续时长回放，不做任何修改 | 保留演奏的原始表情和节奏变化 |
| **量化到节拍** | 将每个风格的持续时长量化到最近的节拍单位（1/4、1/8、1/16拍），由BPM决定实际时长 | 需要与节拍严格对齐的场景 |
| **统一时长** | 所有风格使用相同的固定持续时长（可设为1拍、2拍等），忽略录制时的实际时长 | 追求整齐划一的节奏效果 |

量化到节拍的工作原理：
```
BPM = 128, 节拍精度 = 1/4拍
一拍的时长 = 60 / 128 = 0.46875秒
1/4拍时长 = 0.46875 / 4 = 0.117秒

录制持续时长: 0.35秒
量化: 0.35 / 0.117 ≈ 2.99 → 取整为 3 → 实际回放时长 = 3 × 0.117 = 0.351秒
```

回放控制：
- 回放时播放头按录制的时间线自动前进，依次切换风格
- 支持BPM调节（回放时改变BPM，所有持续时长按比例缩放）
- 保存为预设文件（JSON格式，包含事件列表和BPM信息）
- 支持循环回放
- 回放时可叠加实时手动演奏（叠加模式）

**模式三：自动切换模式**

以节拍为基本时间单位自动切换风格，每段播放时长随机在1-4拍之间，确保切换节奏与音乐节拍严格同步：

- **每段播放时长 = 随机节拍数 × 单拍时长**：
  - 随机节拍数：每次切换前从 **1拍、2拍、3拍、4拍** 中随机选择
  - 单拍时长：`60 / BPM` 秒
  - 示例：BPM=128，随机到3拍 → 播放时长 = (60/128) × 3 = **1.406秒**
- **随机模式设置**：
  - 完全随机：每次独立随机1-4拍
  - 加权随机：可设置各拍数权重（如4拍出现概率更高）
  - 固定模式：指定固定拍数（关闭随机）
- **BPM来源**：
  - 手动输入BPM值
  - 自动检测（基于已加载的音频采样）
  - MIDI时钟同步（可选扩展）
- **切换顺序**：
  - 顺序循环：01→02→03→...→11→01→...
  - 随机切换：每次随机选择一个风格（可排除当前风格）
  - 自定义序列：用户指定切换顺序
- **切换过渡效果**：硬切、淡入淡出、交叉淡化、滤波扫频
- **节拍指示器**：界面显示当前节拍位置和下次切换倒计时

### 2.2 音频处理功能

#### 2.2.1 实时效果器
- **低通/高通滤波器**：实时调节音色
- **混响（Reverb）**：添加空间感
- **延迟（Delay）**：回声效果
- **失真（Distortion）**：饱和效果
- **合唱（Chorus）**：加厚音色

#### 2.2.2 切换过渡效果
- **硬切（Hard Cut）**：立即切换
- **快速淡入淡出（Fade）**：50-200ms淡变
- **交叉淡化（Crossfade）**：两音频重叠过渡
- **滤波扫频（Filter Sweep）**：通过滤波器过渡

#### 2.2.3 输出处理
- **主输出**：立体声输出
- **可视化**：实时频谱显示、波形显示
- **录音功能**：录制演奏输出为WAV/MP3

### 2.3 辅助功能

#### 2.3.1 预设管理
- 保存/加载键盘映射配置
- 保存/加载效果器参数
- 预设分类管理

#### 2.3.2 音频预览
- 单独预览每个键位的音频
- 波形预览显示
- 音频信息展示（时长、采样率、比特率）

#### 2.3.3 快捷键
- 空格键：停止所有播放
- ESC键：紧急静音
- 数字键1-9：快速切换预设
- Ctrl+S：保存当前配置

## 3. 非功能需求

### 3.1 性能需求
- **启动时间**：< 3秒
- **音频加载时间**：< 2秒（11个文件）
- **按键延迟**：< 20ms（端到端）
- **CPU占用**：< 15%（空闲时），< 50%（满复音演奏时）
- **内存占用**：< 500MB

### 3.2 兼容性需求
- **操作系统**：Windows 10/11、macOS 10.15+、Linux Ubuntu 20.04+
- **音频输出**：支持DirectSound、ASIO（Windows）、Core Audio（macOS）
- **最低硬件**：4GB RAM，双核CPU
- **推荐硬件**：8GB RAM，SSD，独立声卡

### 3.3 用户体验需求
- 界面语言：中文
- 界面风格：暗色主题，类似DAW（数字音频工作站）
- 响应速度：所有操作即时反馈
- 错误处理：友好的错误提示，自动恢复

### 4.2 核心模块说明

#### 4.2.1 采样管理器 (SampleManager)
- 职责：管理11个音频采样的加载、缓存和访问
- 预加载：程序启动时将所有采样加载到内存
- 格式转换：统一转换为内部处理格式（32位浮点PCM）
- 时间对齐验证：加载时检查所有采样时长是否一致，不一致时给出警告
- 按位置读取：支持从任意时间点读取指定长度的音频数据

#### 4.2.2 播放头引擎 (PlayheadEngine)
- 职责：管理全局播放头的位置和状态
- 状态管理：播放、暂停、停止、跳转
- 位置追踪：实时记录当前播放头位置（精确到采样级别）
- 速度控制：支持变速播放（0.5x - 2.0x）
- 循环控制：到达末尾后是否循环回到开头
- 输出：当前播放头位置、当前应该输出的音频数据段

#### 4.2.3 风格切换器 (StyleSwitcher)
- 职责：根据按键输入切换当前输出的风格
- 输入：来自键盘/MIDI的风格切换指令
- 切换逻辑：更新当前活跃风格编号，通知播放头引擎从新风格的对应位置读取数据
- 过渡处理：切换时调用过渡处理器进行交叉淡化，避免爆音
- 状态输出：当前活跃的风格编号、切换历史

#### 4.2.4 效果处理器 (EffectProcessor)
- 职责：实时音频效果处理
- 链式结构：支持多个效果串联
- 零延迟处理：使用低延迟算法

#### 4.2.5 主输出 (MasterOutput)
- 职责：将处理后的音频数据输出到音频设备
- 增益控制：防止削波失真
- 电平表：实时显示输出电平

## 6. 技术实现方案

### 6.1 技术栈选择（推荐方案：Rust + Tauri + Web）

> **说明**：本节包含完整的技术栈选型分析，包括候选方案对比和最终推荐。

#### 6.1.1 核心技术挑战

本项目不是普通的"播放器"，而是一个**实时音频采样器**，存在以下关键技术难点：

| 挑战 | 难度 | 说明 |
|------|:----:|------|
| **低延迟音频输出** | ★★★ | 按键到发声 <20ms，需要直接操作音频缓冲区 |
| **共享时间线引擎** | ★★☆ | 11条音轨对齐、播放头精确到采样级别 |
| **实时DSP效果器** | ★★★ | 滤波器/混响/延迟需要在音频回调中逐样本计算 |
| **全局键盘捕获** | ★★☆ | 即使窗口失焦也要响应按键 |
| **实时波形可视化** | ★★☆ | 60fps刷新，不能阻塞音频线程 |
| **MP3编码导出** | ★☆☆ | 非实时操作，无严格性能要求 |

**最核心的矛盾**：GUI渲染（需要主线程）vs 音频处理（需要实时性、不能被GC/UI卡顿打断）。

#### 6.1.2 候选方案对比

| 方案 | 技术栈 | 延迟预估 | 优势 | 劣势 | 结论 |
|------|--------|:--------:|------|------|------|
| **A. Python** | PyQt6 + sounddevice + numpy | 30-50ms | 开发最快、生态丰富 | **GIL导致延迟不稳定**、打包体积大、启动慢 | 原型可行，不适合产品 |
| **B. Rust + Tauri** ⭐推荐 | Rust (cpal) + Tauri + Web前端 | **5-15ms** | **零GC、延迟稳定**、打包极小(~5MB)、前端代码可复用 | Rust学习曲线陡峭 | **最佳方案** |
| **C. Electron** | Electron + Web Audio API | 10-25ms | 纯Web技术、开发者多 | 内存占用高(~150MB)、全局键盘捕获难、偶发GC卡顿 | 次优，适合前端团队 |
| **D. JUCE/C++** | JUCE框架 (C++) | 1-5ms | 专业音频工业标准 | 开发效率极低、GUI丑陋、学习成本极高 | 适合专业音频公司 |

#### 6.1.3 推荐方案详解（Rust + Tauri + Web）

**架构设计**：


**技术选型表**：

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| **开发语言** | Rust 1.70+ | 零成本抽象、无GC、内存安全 |
| **桌面框架** | Tauri v2 | 轻量级、小体积、系统原生WebView |
| **音频引擎** | cpal | Rust生态最成熟的跨平台音频库 |
| **DSP处理** | 手写Rust | 实时滤波器/混响/延迟，SIMD优化 |
| **GUI前端** | HTML/CSS/JS | 直接复用现有Demo，60fps动画 |
| **前后端通信** | Tauri IPC / WebSocket | 微秒级延迟 (~0.1ms) |
| **音频解码** | symphonia | Rust纯实现MP3/FLAC解码 |
| **音频编码** | hound | WAV编码，导出录制文件 |
| **配置序列化** | serde + RON/JSON | 预设文件格式 |
| **全局键盘** | rdev (Rust) | 跨平台全局键盘监听 |

**为什么选择Rust + Tauri？**

1. **延迟达标**：Rust音频线程完全无GC，延迟稳定在5-15ms，远优于Python的30-50ms
2. **打包极小**：Tauri应用约5MB，Electron约80MB，Python约100MB+
3. **前端复用**：Phase 0的HTML Demo可以直接嵌入Tauri，几乎不用修改
4. **专业级DSP**：Rust的`unsafe`块可以做SIMD优化，性能接近C++
5. **现代工具链**：Cargo包管理、Clippy静态检查、原生跨平台编译

**为什么不选其他方案？**

- **Python**：GIL导致延迟不稳定，不适合实时音频产品
- **Electron**：内存占用大、全局键盘捕获难、V8偶发GC卡顿
- **JUCE/C++**：开发效率太低，GUI太丑，学习成本过高

#### 6.1.4 备选方案（Electron + Web Audio API）

如果团队全是前端背景，可选用此方案：

| 组件 | 技术选型 |
|------|---------|
| 框架 | Electron + React/Vue |
| 音频引擎 | Web Audio API (AudioWorklet) |
| DSP | AudioWorkletProcessor (独立线程) |
| 全局键盘 | iohook (native模块) |
| 打包 | electron-builder |

**注意**：Web Audio API的AudioWorklet有128样本固定缓冲区，理论最低延迟2.7ms，但实际受Electron V8 GC影响，偶发卡顿。

#### 6.1.5 开发阶段规划

| 阶段 | 内容 | 技术栈 | 产出 |
|------|------|--------|------|

| **Phase 1** | Rust音频引擎核心 | cpal + 手写DSP | 独立音频进程，低延迟播放 |
| **Phase 2** | Tauri集成前后端 | Tauri IPC | 完整桌面应用框架 |

#### 6.2.1 共享时间线音频回调
```python
# 音频回调函数（伪代码）—— 共享时间线模型 + 循环回跳
def audio_callback(out_data, frame_count, time_info, status):
    global playhead_pos, active_style, is_playing
    
    output_buffer = np.zeros((frame_count, 2), dtype=np.float32)
    
    if is_playing and active_style is not None:
        sample = samples[active_style]
        start_sample = int(playhead_pos * sample_rate)
        end_sample = start_sample + frame_count
        
        if end_sample <= len(sample):
            # 正常读取
            output_buffer = sample[start_sample:end_sample]
            playhead_pos += frame_count / sample_rate
        else:
            # 到达末尾：检测是否有按键保持按下
            remaining = len(sample) - start_sample
            output_buffer[:remaining] = sample[start_sample:]
            
            if any_key_pressed and loop_mode != 'off':
                # 有按键按下 + 循环启用 → 回跳到循环起点继续
                loop_start_sample = int(loop_start_pos * sample_rate)
                needed = frame_count - remaining
                output_buffer[remaining:] = sample[loop_start_sample:loop_start_sample + needed]
                playhead_pos = loop_start_pos + needed / sample_rate
            else:
                # 无按键按下 或 循环关闭 → 停止播放
                is_playing = False
                playhead_pos = len(sample) / sample_rate  # 停在末尾
    
    # 应用效果器
    output_buffer = effect_chain.process(output_buffer)
    
    # 输出到设备
    out_data[:] = output_buffer.tobytes()
    return (out_data, pyaudio.paContinue)
```

```python
def on_key_press(key):
    key_map = {
        'a': 0, 's': 1, 'd': 2, 'f': 3, 'g': 4,
        'h': 5, 'j': 6, 'k': 7, 'l': 8, ';': 9, "'": 10

    if key.char in key_map:
        new_style = key_map[key.char]
        switch_to_style(new_style)  # 切换风格，播放头不回退

def switch_to_style(new_style):
    global active_style, is_playing
    
    if trigger_mode == 'gate':
        # 门控模式：切换到新风格，播放头继续前进
        if not is_playing:
            is_playing = True  # 首次按键，开始播放
        active_style = new_style
    elif trigger_mode == 'trigger':
        # 触发模式：切换后持续播放直到结束
        is_playing = True
        active_style = new_style

def on_key_release(key):
    global is_playing
    
    if trigger_mode == 'gate':
        # 门控模式：松开键，暂停播放（播放头保持位置）
        is_playing = False
```

#### 6.2.3 实时效果器实现
```python
# 简单的低通滤波器（一阶）
class LowPassFilter:
    def __init__(self, cutoff_freq, sample_rate):
        self.rc = 1.0 / (2 * np.pi * cutoff_freq)
        self.dt = 1.0 / sample_rate
        self.alpha = self.dt / (self.rc + self.dt)
        self.y_prev = 0
    
    def process(self, input_signal):
        output = np.zeros_like(input_signal)
        for i in range(len(input_signal)):
            self.y_prev += self.alpha * (input_signal[i] - self.y_prev)
            output[i] = self.y_prev
        return output
```

**总工期预估**：约5周（不含Phase 0）

### 8.2 性能优化策略

| 优化点 | 策略 | 技术实现 |
|--------|------|----------|
| **启动速度** | 延迟加载非必要资源 | 采样按需解码，预设懒加载 |
| **音频延迟** | 使用ASIO/Core Audio，优化缓冲区大小 | cpal配置低延迟模式，缓冲区128-512样本 |
| **DSP性能** | SIMD向量化、避免内存分配 | Rust `packed_simd` 或 `std::simd` |
| **UI响应** | 音频线程与UI线程分离 | Rust音频回调永不阻塞，IPC异步通信 |
| **内存占用** | 音频数据预加载、复用缓冲区 | 采样一次加载到`Vec<f32>`，效果器使用环形缓冲区 |
| **打包体积** | 启用LTO、strip符号 | `cargo build --release` + `strip` |

### 8.3 Rust开发注意事项

**音频回调中的禁忌**（会导致爆音/卡顿）：
- ❌ 在音频回调中使用`unwrap()`或`panic!`
- ❌ 在音频回调中分配内存（`Vec::new()`、`Box::new()`）
- ❌ 在音频回调中打印日志或访问文件
- ❌ 在音频回调中持有锁（`Mutex::lock()`）

**正确做法**：
- ✅ 预分配所有缓冲区，音频回调只进行读写
- ✅ 使用`try_lock()`或`RwLock`的读锁，失败时跳过而非阻塞
- ✅ 通过channel将事件发送到UI线程处理
- ✅ 使用`ringbuf`等无锁环形缓冲区

**推荐Rust学习路径**：
1. [Rust官方Book](https://doc.rust-lang.org/book/)（前10章）
2. [Rust Audio Learn](https://github.com/RustAudio/learn)（音频特化）
3. [cpal examples](https://github.com/RustAudio/cpal/tree/master/examples)（音频引擎参考）
